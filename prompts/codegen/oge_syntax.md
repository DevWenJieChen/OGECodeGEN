- ### 1. Initialization (fixed pattern)

  1. You must first import and initialize OGE:

  ```
  import oge
  oge.initialize()
  service = oge.Service()
  ```

  2. All subsequent data acquisition and processing operations must be completed through the `service` object.

  ------

  ### 2. Obtain Data Objects (Coverage / CoverageCollection / Feature / FeatureCollection)

  #### 2.1 Obtain a single Coverage (most common)

  Use `getCoverage` to obtain a Coverage object:

  ```
  cov = service.getCoverage(
      coverageID="<coverage_id>",
      productID="<product_id>"
  )
  ```

  Constraints:

  - coverageID: string literal, the identifier of a scene/image (for example: LC08_L1GT_121039_20240325_20240403_02_T2)
  - productID: string literal, the product name (for example `"LC08_C02_L1"`)
  - Return type: `Coverage`

  #### 2.2 Obtain a CoverageCollection (range + time)

  When data must be read by spatial extent or time range, use `getCoverageCollection`:

  ```
  cov_col = service.getCoverageCollection(
      productID="<product_id>",
      bbox=[minLon, minLat, maxLon, maxLat],
      datetime=["<start_date>", "<end_date>"]
  )
  ```

  Parameter descriptions (must be strictly followed):

  - productID: string, the product name (for example `"LC08_C02_L1"`)
  - bbox: `List[float]`, spatial extent `[minLon, minLat, maxLon, maxLat]`
  - datetime: `List[str]`, time range, for example `["2024-01-01 00:00:00", "2024-03-01 00:00:00"]`

  Important type constraints (hard rules):

  - The return type of `getCoverageCollection` is `CoverageCollection`
  - `CoverageCollection` cannot directly use any `Coverage` operator
  - `bboxCrs` may use the default; do not specify it unless necessary

  #### 2.3 Obtain a single Feature (vector feature)

  Use Feature when the task needs to read or construct a single vector feature.

  **Method A: Read a single Feature from a platform data source**

  ```
  feature = service.getFeature(featureId="<feature_id_or_asset_path>")
  ```

  Constraints:

  - featureId: string literal (for example `"myData/EasternChina_PopulationAging_Vector.geojson"`)
  - Return type: `Feature`

  > Note: Feature data reading capabilities are still evolving and may support more data sources and parameter forms in the future. Currently, `getFeature(featureId=...)` is the main method.

  **Method B: Build from a geometry object (commonly used for examples/temporary data)**

  First build Geometry, then use `Feature.loadFromGeometry` to generate Feature:

  ```
  geom = service.getProcess("Geometry.Polygon").execute(coords, "EPSG:4326")
  feature = service.getProcess("Feature.loadFromGeometry").execute(geom, "{a:10}")
  ```

  Constraints:

  - Geometry construction is generated by `Geometry.*` operators (such as `Geometry.LineString` / `Geometry.Polygon`)
  - CRS is usually passed explicitly (for example `"EPSG:4326"`)
  - The attribute parameter of `Feature.loadFromGeometry` is often passed as a string (for example `"{a:10}"`)

  #### 2.4 Obtain a FeatureCollection (feature set)

  Current common method: build a FeatureCollection from a list of Features.

  A FeatureCollection represents a set of vector features and is commonly used for point sets, line sets, polygon sets, uploaded GeoJSON data, facility data, road data, administrative-region data, etc.

    Common Method A: Read FeatureCollection from an uploaded vector file

    fc = service.getProcess("FeatureCollection.loadFeatureCollectionFromUpload").execute(
        "myData/example.geojson",
        "EPSG:4326"
    )

    Applicable scenarios:
    - `data_info` or user input provides uploaded files such as `myData/*.geojson`;
    - The task involves complete feature collections such as roads, facility points, administrative regions, or multi-year vector data;
    - Subsequent steps need collection operators such as `FeatureCollection.filterMetadata`, `FeatureCollection.size`, `FeatureCollection.intersection`, or `FeatureCollection.difference`.

    Common Method B: Read FeatureCollection from a platform data source

    fc = service.getFeatureCollection(featureId="<feature_collection_id_or_asset_path>")

    Applicable scenarios:
    - The platform provides a FeatureCollection data asset;
    - The input itself is a vector dataset, not a single Feature.

    Common Method C: Temporarily build a FeatureCollection from several Features

    fc = service.getProcess("FeatureCollection.loadFromFeatureList").execute([feature1, feature2])

    Applicable scenarios:
    - Multiple Features are constructed temporarily in the code;
    - Multiple single features need to be combined into a collection for subsequent analysis.
    - Do not treat `loadFromFeatureList` as the default method for reading an uploaded GeoJSON dataset.

  Constraints:

  - Input: `List[Feature]`
  - Output: `FeatureCollection`

  > Note: FeatureCollection reading/building methods are also evolving and may support direct range reading, data-source filtering, etc. in the future. Currently, "building from a Feature list" is the main method.

  ------

  ### 3. CoverageCollection → Coverage (required intermediate step)

  If you want to process a `CoverageCollection` with `Coverage` operators, you must first convert it to `Coverage`.

  The only allowed method is to use `CoverageCollection.mosaic` to mosaic all images into a single Coverage image:

  ```
  cov = service.getProcess("CoverageCollection.mosaic").execute(cov_col)
  ```

  Constraints (hard):

  - Input: `CoverageCollection`
  - Output: `Coverage`
  - Only after mosaic is completed may you continue using `Coverage` operators

  Incorrect example (forbidden):

  ```
  ndvi = service.getProcess("Coverage.normalizedDifference").execute(cov_col, ["B5", "B4"])
  ```

  Correct example:

  ```
  cov = service.getProcess("CoverageCollection.mosaic").execute(cov_col)
  ndvi = service.getProcess("Coverage.normalizedDifference").execute(cov, ["B5", "B4"])
  ```

  #### (Supplement) Relationship between FeatureCollection and Feature (general rule)

  - The relationship between `Feature` and `FeatureCollection` is **similar** to the relationship between `Coverage` and `CoverageCollection`: the former is a single object, and the latter is a collection.
  - However, currently there is **no unified hard rule that FeatureCollection must first be converted to Feature**:
    - Many operators are `FeatureCollection.*` (such as filter/aggregate/geoHash) and act directly on collections;
    - Some operators are `Feature.*` (such as intersects) and require a single Feature input.
  - If you really need to obtain a single feature or statistical result from a collection, use collection operators such as `FeatureCollection.aggregate*` / `aggregateFirst` to produce the required output (which may be a value or an object for further visualization/rasterization), rather than assuming a fixed "mosaic-like" conversion exists.

  ------

  ### 4. Obtain a Process Instance

  Use `getProcess` to obtain a processing operator:

  ```
  proc = service.getProcess("<process_id>")
  ```

  Constraints:

  - process_id: string literal, for example:
    - Coverage-related: `"Coverage.selectBands"`, `"Coverage.focalMean"`, `"Coverage.normalizedDifference"`, `"CoverageCollection.mosaic"`
    - Feature/Geometry-related: `"Geometry.LineString"`, `"Geometry.Polygon"`, `"Feature.loadFromGeometry"`, `"Feature.intersects"`, `"FeatureCollection.filterMetadata"`, `"FeatureCollection.geoHash"`, `"FeatureCollection.aggregateCount"`

  ------

  ### 5. Execute Processing (Process.execute)

  #### 5.1 Basic pattern (recommended)

  ```
  out = service.getProcess("<process_id>").execute(arg1, arg2, ...)
  ```

  #### 5.2 Parameter rules (very important)

  1. The first argument is usually the **main data object** (Coverage / Feature / CoverageCollection / FeatureCollection)
  2. Subsequent arguments must be given in the order defined by the operator
  3. Use positional arguments; do not use keyword arguments

  Example (Coverage):

  ```
  b3 = service.getProcess("Coverage.selectBands").execute(ls8, ["B3"])
  mean = service.getProcess("Coverage.focalMean").execute(b3, "square", 1)
  ```

  Example (Feature/FeatureCollection):

  ```
  fc2 = service.getProcess("FeatureCollection.filterMetadata").execute(fc, "a", "greater_than", "11")
  flag = service.getProcess("Feature.intersects").execute(feature1, feature2, "EPSG:4326")
  ```

  Incorrect example (forbidden):

  ```
  service.getProcess("Coverage.focalMean").execute(cov=b3, kernel="square", radius=1)
  ```

  ------

  ### 6. Visualization and Export

  #### 6.1 Visualization parameters (Coverage raster)

  ```
  vis_params = {
      "min": <number>,
      "max": <number>,
      "palette": ["color1", "color2", ...]
  }
  ```

  The palette color order must strictly map from low values to high values, and the color choices should match common land-cover semantics (for example, water as blue and vegetation as green) and avoid abrupt visual choices.

  #### 6.2 Map display

  - Coverage (raster):

  ```
  cov.styles(vis_params).getMap("<layer_name>")
  ```

  - Feature / FeatureCollection (vector; currently commonly styled as a color list):

  ```
  feature_or_fc.styles(["#FF0000"]).getMap("<layer_name>")
  ```

  > Note: Feature styling capabilities are still evolving. Current examples mainly use color lists. More symbolization parameters may be supported in the future, but the call pattern will still follow `.styles(...).getMap(...)`.

  #### 6.3 Export (without display)

  ```
  cov.styles(vis_params).export("<layer_name>")
  ```

  (If Feature/FeatureCollection supports export, follow the same `.styles(...).export(...)` pattern.)

  ------

  ### 7. Map View Setting (very important; strongly recommended)

  In online computation / map display mode, you must explicitly set the map view:

  ```
  oge.mapclient.centerMap(lon, lat, zoom)
  ```

  Key notes:

  - Whether data exists depends on the latitude/longitude range corresponding to the current map window
  - If the zoom is too high or the center point is outside the data coverage area, no data may be read
  - The initial zoom should not be too high (for example, 6-8)
  - When the range is uncertain, lower the zoom to expand the coverage window

  ------

  ### 8. Hard Constraints for LLM-Generated Code (strictly follow)

  1. You must first write:

  ```
  import oge
  oge.initialize()
  service = oge.Service()
  ```

  2. Obtain data objects:

  - Coverage:

  ```
  cov = service.getCoverage(coverageID="...", productID="...")
  ```

  Or range-based reading:

  ```
  cov_col = service.getCoverageCollection(productID="...", bbox=[...], datetime=[...])
  ```

  - Feature (vector):

  ```
  feature = service.getFeature(featureId="...")
  ```

  Or build from geometry (temporary/example):

  ```
  geom = service.getProcess("Geometry.*").execute(..., "EPSG:4326")
  feature = service.getProcess("Feature.loadFromGeometry").execute(geom, "{...}")
  ```

  - FeatureCollection (collection):

  ```
  # If the input is an uploaded GeoJSON / converted Shapefile file / multi-feature vector dataset,
  # prefer reading it as FeatureCollection:
  fc = service.getProcess("FeatureCollection.loadFeatureCollectionFromUpload").execute(
    "myData/example.geojson",
    "EPSG:4326"
  )

  # If the platform provides a FeatureCollection data asset, you can also use:

  fc = service.getFeatureCollection(featureId="...")

  # If multiple Features are temporarily constructed in code, then use:

  fc = service.getProcess("FeatureCollection.loadFromFeatureList").execute([feature1, feature2, ...])
  ```

  3. Use `getProcess` to obtain a Process:

  ```
  proc = service.getProcess("...")
  ```

  4. Execute processing with positional arguments:

  ```
  out = proc.execute(arg1, arg2, ...)
  # or
  out = service.getProcess("...").execute(arg1, arg2, ...)
  ```

  5. Hard type rules:

  - Clearly distinguish `Coverage` from `CoverageCollection`; `CoverageCollection` must be mosaicked before `Coverage` operators can be used
  - Clearly distinguish `Feature` from `FeatureCollection`:
    - `Feature.*` operators take single features
    - `FeatureCollection.*` operators take feature collections
    - Do not assume a unified mandatory FeatureCollection→Feature conversion workflow; choose inputs according to operator type

  6. When displaying or exporting maps, use:

  - Coverage:

  ```
  cov.styles(vis_params).getMap("name")
  # or
  cov.styles(vis_params).export("name")
  ```

  - Feature / FeatureCollection (current common usage):

  ```
  obj.styles(["#RRGGBB"]).getMap("name")
  ```

  7. Try not to use `try/except`; it triggers the safety-protection mechanism and is not allowed to execute (the OGE platform execution environment is not a normal Python interpreter environment).

  Constraint summary:

  - Initialization code must be complete and in a fixed order
  - Coverage/CoverageCollection and Feature/FeatureCollection must be distinguished
  - CoverageCollection must be mosaicked before Coverage operators can be used
  - `execute` uses positional arguments, not keyword arguments
  - For map display or online computation, set a reasonable `centerMap`
